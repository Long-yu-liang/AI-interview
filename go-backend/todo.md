# 项目优化与重构建议清单

本文档汇总了对 `ai-gozero-agent` 项目的深度审查结果，涵盖架构设计、AI 最佳实践、工程规范及安全性等维度的优化建议。

## 一、核心架构与服务问题

### 1. MCP 服务 (PDF 解析) 重构
目前 `mcp` 目录下的服务实为 "PDF 解析服务"，与 Anthropic 的 Model Context Protocol 命名冲突且实现存在隐患。
    
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P1** | **磁盘 I/O 瓶颈** | 服务端将流数据写入本地临时文件，高并发下磁盘成为瓶颈，且容器化部署下容易丢失。 | 改为**内存流式处理** (`io.Pipe` / `bytes.Buffer`)，避免落盘。 |
| **P1** | **依赖商业库** | 使用了 `unidoc/unipdf` (商业库)，未配置 Key 会导致解析失败或水印，存在合规风险。 | 替换为开源库 (如 `rsc/pdf`, `ledongthuc/pdf`) 或云厂商文档解析 API。 |
| **P2** | **gRPC 反模式** | 响应中包含 `error` 字符串字段，而非使用 gRPC 标准状态码。 | 移除 `error` 字段，使用 `google.golang.org/grpc/status` 返回标准错误码。 |
| **P2** | **单向阻塞** | 客户端流式上传，服务端全量接收后才处理，大文件延迟高 (TTFB)。 | 改为**双向流**或服务端流，边接收边解析边返回。 |
| **P3** | **命名混淆** | `mcp` 易与标准协议混淆。 | 重命名为 `pdf-service` 或 `doc-processor`。 |

#### 具体重构需求与验收标准 (Spec & Acceptance Criteria)
结合市场主流 RAG 需求，本服务的重构目标不仅仅是修复 Bug，而是升级为**生产级文档解析服务**。

**1. 功能维度 (Functionality)**
*   **Markdown 结构化输出**: 解析结果不应是纯文本流，而是保留文档结构的 Markdown (标题、列表、表格)。LLM 对 Markdown 的理解能力远强于纯文本。
*   **智能分块 (Smart Chunking)**: 基于 Markdown 结构（如按 H2/H3 标题）进行语义分块，而非机械的字符数截断。
*   **表格与图片占位**: 识别文档中的表格并转为 Markdown Table；图片需保留占位符 `[Image: page_1_img_1]` 或提取 OCR (可选阶段)。

**2. 工程维度 (Engineering)**
*   **去临时文件化**: 彻底移除本地磁盘写入逻辑，实现全内存/流式管道处理。
*   **双向流式协议**: 改造 gRPC 接口为 `Bidi Streaming`。客户端上传文件流，服务端实时推送解析进度（百分比）和解析完成的片段（Chunk），大幅降低首字延迟。
*   **标准错误处理**: 废弃 `error` 字符串字段，使用 `google.golang.org/grpc/status` 定义业务错误码（如 `INVALID_FILE_FORMAT`, `PARSING_TIMEOUT`）。

**3. 选型建议 (Tech Stack)**
*   **移除 UniPDF**: 替换为无版权风险的方案。
    *   *方案 A (轻量)*: 使用 `ledongthuc/pdf` (纯 Go) 提取文本，自行实现简单的 Layout 分析转 Markdown。
    *   *方案 B (生产级)*: 接入大模型厂商的文档解析 API (如阿里的 DocMind 或 ByteDance 的 Coze API)，以 Adapter 模式集成，确保解析效果达到商业级标准。

### 2. 向量检索与 RAG 优化
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P0** | **伪向量检索** | 使用 JSONB 存向量 + 文本相似度排序，丧失语义检索能力。 | 迁移至 **pgvector**，使用 HNSW/IVFFlat 索引和余弦相似度 (`<=>`)。 |
| **P2** | **历史无检索** | 聊天历史生成了向量但仅按时间拉取，浪费 Token 且丢失上下文细节。 | 实现**混合检索**：最近 N 条 (时序) + Top K 相关 (语义) 历史消息。 |

## 二、AI 应用开发最佳实践

### 1. Prompt 与上下文管理
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P1** | **Prompt 硬编码** | System Prompt 写死在 Go 代码中，无法动态调优和版本管理。 | 抽离 Prompt 到配置文件、数据库或 Prompt 管理平台；支持热更新。 |
| **P1** | **上下文无 Token 感知** | 仅按条数 (10条) 截断，可能撑爆上下文窗口或丢失关键信息。 | 引入 `tiktoken` 计算 Token 数，基于 **Token 窗口** (如 4k/8k) 进行滑动截断。 |
| **P2** | **状态机简陋** | 面试状态仅依赖 Redis 字符串，缺乏严格流转约束。 | 设计结构化状态机，并在 Prompt 中注入当前状态的明确指令约束。 |

### 2. 安全与配置
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P0** | **API Key 泄露** | `chat.yaml` 中明文配置 Key，代码库泄露即导致密钥失窃。 | 必须通过**环境变量**注入 (如 `OPENAI_API_KEY`)，配置文件仅留占位符。 |

## 三、工程规范与代码质量 (新发现)

### 1. 健壮性与错误处理
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P0** | **错误被吞** | `ChatHandler` 中参数解析失败或业务错误直接 `return`，前端无响应直到超时。 | 必须返回 HTTP 4xx/5xx 状态码及错误信息 (JSON)。 |
| **P1** | **SSE 实现脆弱** | 手动拼接字符串 `data: %s` 并手动转义，易出错且格式不标准。 | 封装标准的 SSE 写入器，使用 JSON 序列化数据对象，确保特殊字符安全。 |
| **P2** | **配置缺失 Env 标签** | `config.go` 中结构体缺少 `json:",env=..."` 标签，无法自动加载环境变量。 | 为敏感配置项添加 `env` 标签，支持从环境自动注入。 |

### 2. 测试与部署
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P0** | **零测试覆盖** | 项目中无任何 `*_test.go` 文件，重构和上线风险极高。 | 补充核心 Logic 的单元测试 (`mock` 外部依赖)，特别是 `ChatLogic` 和 `VectorStore`。 |
| **P2** | **Dockerfile 基础镜像** | 使用 `alpine` 但未安装 `ca-certificates` 和 `tzdata`。 | 增加 `RUN apk add --no-cache ca-certificates tzdata`，确保 HTTPS 和时区正常。 |

### 3. 代码规范
| 优先级 | 问题点 | 风险描述 | 优化建议 |
| :--- | :--- | :--- | :--- |
| **P3** | **Handler 逻辑过重** | `ChatHandler` 包含 PDF 处理、SSE 循环等大量逻辑。 | 将业务逻辑下沉到 `Logic` 层，Handler 仅负责参数解析和响应格式化。 |
